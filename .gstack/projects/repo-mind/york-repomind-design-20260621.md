> This design narrows a 25-section platform spec (idea.md) into a shippable
> open-source wedge. The full platform — SQLite draft layer, local Web UI,

## Problem Statement

AI-assisted development (Cursor, Claude Code, Codex, OpenClaw) is now common,
but project knowledge is scattered across README, markdown, Notion, Jira,
Confluence, task comments, private agent instructions, verbal agreements, and
stale architecture decisions. As a result, agents lose context, don't know
current project rules, invent solutions, misread architecture, repeat old
mistakes, and conflate drafts with approved knowledge.

RepoMind's thesis: create one structured project-knowledge layer inside the
repository that is readable by developers, versioned through Git, and —
crucially — queryable by AI agents through MCP instead of ingesting whole
files.

The user is running gstack, which already partially solves this (CLAUDE.md
skill routing, `~/.gstack/projects/` design docs, learnings, decisions, brain
cache). That is the real closest competitor, not Notion.

## What Makes This Cool

The "whoa" is zero-effort ambient agent knowledge: drop RepoMind into a repo,
point Claude Code at the MCP, and the agent instantly knows your glossary,
ADRs, and open questions — without a human curating a prompt. The magic is
"the agent just knows."

Two complementary cool moments, in order:
1. **Agent queries instead of ingests.** `search_docs` / `get_glossary_term` /
   `explore_graph` return the one relevant doc, not 50KB of markdown. Less
   context in, fewer hallucinations out. This is the differentiator plain
   markdown + CLAUDE.md structurally cannot match — they dump, they don't
   query.
2. **Agent proposes, human merges.** A light `create_draft` tool lets an agent
   open a git branch with a proposed doc (glossary term, Open Question, ADR);
   a human reviews and merges via normal PR. No database, no status machine —
   just git. This is the one capability plain markdown + CLAUDE.md structurally
   cannot do at all: CLAUDE.md is human-authored, full stop.

Positioning, borrowed from the cross-model read: **RepoMind is "Context7 for
your own repo's institutional knowledge"** — Context7 serves public library
docs to coding agents via MCP; RepoMind serves your private project knowledge
to your agents via MCP, query-by-intent instead of fetch-by-library.

## Constraints

- Open source / research goal: adoption and contributor mindshare are the
  currency, not revenue. No auth, no hosting, no team RBAC in v1.
- Local-only. The CLI and MCP server run on the developer's machine against
  the cwd repo. No cloud. Zero network at runtime (install uses npm as normal
  for npx; `npx -y` pulls latest at install time, not at query time).
- Install-to-first-whoa must be under 5 minutes: install, `init`, paste a
  one-line MCP config, ask the agent a project question.
- No persistent state in v1 — no SQLite, no draft store, no status machine.
  The index is derived at runtime from files + frontmatter and held in an
  in-process memory map, invalidated by file-mtime check, for the MCP server's
  lifetime. No cross-process cache. (A SQLite *read index* is a v1.1 option,
  not v1.)
- TypeScript + Node, distributed via npx. v1 ships zero native deps; note
  Approach B reintroduces a native SQLite binding in v1.1.
- v1 storage format is plain Markdown + YAML frontmatter. MDX/Markdoc are
  deferred to v1.1 (they add a parser dependency and editor complexity that
  only earn their keep once the Web UI lands).
- Generic templates win adoption; game-design/lore/economy are example
  templates proving extensibility, not the path to broad adoption.

## Premises

All five agreed during the session. P1 was revised after the cross-model
challenge.

- **P1 (revised): v1 is narrow and MCP-native.** `.project-knowledge/`
  convention + MCP server with **five read-only tools** (`list_docs`,
  `search_docs`, `get_doc`, `get_glossary_term`, `explore_graph`) over plain
  Markdown + YAML frontmatter, PLUS **one write tool** `create_draft` that
  creates a git branch with a proposed doc and returns the branch name. No
  SQLite, no Web UI, no review/publish state machine, no graph engine in v1.
  The draft gate is git-native (agent proposes on a branch, human merges), not
  a SQLite draft layer. `create_draft` ships only after the P2 kill-switch
  passes — it is not load-bearing for the retrieval thesis and must not delay
  the demo.
- **P2: Retrieval is the differentiator that justifies existence.** The
  measurable win is less context + fewer hallucinations vs plain markdown +
  CLAUDE.md. If the A/B demo cannot show that gap, the project has no reason
  to exist. This is the go/no-go kill-switch.
- **P3: The real competitor is plain markdown + CLAUDE.md (and gstack), not
  Notion.** Every feature must survive "can't a tidy `docs/` folder + a good
  CLAUDE.md do this?" If yes, cut it from v1.
- **P4: Distribution is part of the design.** Ships as a CLI (npx) + a
  one-line MCP config. Install-to-first-whoa under 5 minutes is the
  acceptance test.
- **P5: Generic templates drive adoption.** ADR, Feature Spec, Glossary Term,
  Open Question, Agent Instruction are the core document types.
  Game-design/lore/economy are example templates, not the adoption path.

**How the cross-model read changed the design:** It challenged P1's "no review

the one capability plain markdown structurally lacks, at near-zero cost. The
heavy SQLite gate remains deferred. Time-travel queries were noted as a
roadmap highlight, not v1.

## Approaches Considered

### Approach A: Minimal Viable — "MCP-only, frontmatter-as-index"
Single npx CLI + MCP server in TypeScript. `.project-knowledge/` folder of
markdown + YAML frontmatter docs. No index file — the index is derived at
runtime by globbing + parsing frontmatter. Five read tools + `create_draft` +
`init` + `export`. Grep-ranked full-text search; BFS over `related:`
frontmatter for `explore_graph`. Ships with the A/B demo repo as the P2
kill-switch.
- Effort: S. Risk: Low.
- Pros: smallest diff, ships in a weekend, zero persistent state, demo built
  in validates P2 from day one.
- Cons: re-globs + re-parses frontmatter on every call (fine at hundreds of
  docs, slow at thousands); grep-rough ranking, not semantic.
- Reuses: `@modelcontextprotocol/sdk`, `gray-matter`, `fast-glob`,
  `minimatch`.

### Approach B: Ideal Architecture — "SQLite index, files stay source of truth"
Repo files (MDX/YAML) remain the durable published truth; a SQLite *index*
(`.project-knowledge/.state/index.db`) caches parsed frontmatter, search
tokens, and graph edges, rebuilt by `repo-mind index` or file-watch in dev.
MCP tools query SQLite fast; `get_doc` returns file content; `create_draft`
still writes a git branch. SQLite is a cache, not a draft store.
- Effort: M. Risk: Med.
- Pros: scales to thousands of docs with fast structured search and graph
  traversal; index rebuildable from files (cache, not a second source of
  truth); honors the original doc's SQLite instinct used correctly.
- Cons: adds a build/watch step and a cache that can drift stale; reintroduces
  SQLite before the thesis is proven.
- Reuses: `better-sqlite3` / `sql.js`; the file convention and tools from A.

### Approach C: Creative/Lateral — "Context7-clone registry + public export endpoint"
Mirror Context7's distribution model — tool names like `resolve_project_id` /
`get_project_docs` so anyone using Context7 recognizes the pattern, plus the
query-by-intent tools Context7 lacks. Add `repo-mind publish` that pushes a
read-only exported snapshot to a static host (GitHub Pages/Cloudflare) so
external agents (Codex web, ChatGPT) can query your project knowledge without
repo access. RepoMind as "Context7 for private/repo knowledge, with a
queryable public export."
- Effort: L. Risk: High.
- Pros: sharpest market positioning; uniquely serves external/non-CLI agents
  via the public-export endpoint.
- Cons: a static-host + registry path is a different product surface (hosting,
  auth for private repos) that turns an open-source CLI into a service-op and
  distracts from the <5-min local whoa — too much scope for v1.
- Reuses: Context7's MCP tool shapes; the export pipeline from A/B.

## Recommended Approach

**Approach A — Minimal Viable.**

Rationale mapped to the builder's goal: the stated goal is open-source
adoption measured by a <5-minute whoa, and the P2 kill-switch (the A/B demo
proving retrieval beats plain markdown) must run *before* any architecture
investment. A ships that demo in a weekend with zero persistent state. B is
the right v1.1 once adoption shows scale matters (a repo with thousands of
docs hitting perf walls is the good problem to have). C is a v2 distribution
bet, not a v1 — its hosting/auth surface contradicts the local-only, no-cloud
constraint and would be built only after A's demo proves a developer actually
prefers RepoMind over a tidy `docs/` folder.

## Tool Contracts (v1)

Implementable specs for each tool, so an engineer doesn't have to invent them.

**Frontmatter schema.** Every doc: plain Markdown + YAML frontmatter.
- Required: `type` ∈ {`adr`, `feature-spec`, `glossary-term`, `open-question`,
  `agent-instruction`}; `slug` (unique kebab-case id, enforced by `init` +
  `check`); `status` ∈ {`draft`, `proposed`, `accepted`, `superseded`}.
- Optional: `title`, `tags` (array), `related` (array of slugs — must resolve
  to existing docs, enforced by `check`), `owner`, `updated` (ISO date).
- For `glossary-term` only: the `definition` returned by `get_glossary_term`
  is the first non-empty paragraph of the body.

**Type → directory mapping** (one file per doc, slug-based):
- `adr` → `.project-knowledge/adr/<slug>.md`
- `feature-spec` → `.project-knowledge/specs/<slug>.md`
- `glossary-term` → `.project-knowledge/glossary/<slug>.md`
- `open-question` → `.project-knowledge/open-questions/<slug>.md`
- `agent-instruction` → `.project-knowledge/agents/<slug>.md`

**Knowledge-root discovery (A3).** The server does NOT assume cwd is the repo
root. On startup (and before any tool that touches files), it walks up from
process cwd to the first ancestor containing a `.project-knowledge/` directory,
the way git walks up to `.git`. That ancestor is the *knowledge root*; it is
resolved once and cached for the process lifetime. If none is found, every
tool returns `{found: false}` / empty arrays and `check`/`init` emit a clear
"no .project-knowledge/ found — run `repo-mind init`" message. **(F16)
monorepo:** "nearest wins" — walk-up stops at the *first* ancestor containing
`.project-knowledge/`, so a monorepo with a root `.project-knowledge/` and a
subpackage `.project-knowledge/` scopes the agent to the subpackage when
invoked from it (intentional — subpackage knowledge is more specific;
documented so it isn't mistaken for a bug). This prevents the
silent-empty-result bug when the MCP server is launched from a subpackage.

**Slug & path safety (A1) — security gate.** Every tool that accepts a `slug`
or derives a path (`get_doc`, `explore_graph`, `get_glossary_term` resolution,
`create_draft`) MUST, before any filesystem read or write:
1. Validate the slug against `^[a-z0-9][a-z0-9-]*$` (reject `/`, `..`, `\`,
   spaces, and any non-kebab char). Invalid slug → `{found: false}` (read) or a
   typed error (write), never a path operation.
2. Resolve the path as `path.join(knowledgeRoot, typeDir, slug + ".md")`, then
   `path.resolve` it and assert the result starts with `knowledgeRoot + sep`.
   This catches traversal even if a future code path constructs the slug from
   frontmatter `related:` entries.
3. Only after both checks pass, read or write.
This is the single security invariant of the tool: no read or write may
escape the knowledge root. A dedicated test must exercise `slug` values of
`../x`, `a/../../b`, `..%2f..`, absolute paths, and empty string, asserting
each returns `{found:false}`/error with zero filesystem access outside
`.project-knowledge/`.

**Server lifecycle (A2).** The MCP stdio server is a long-lived child of the
agent. It registers handlers for `SIGTERM`, `SIGINT`, and stdin-close (the MCP
shutdown convention) that:
1. Stop accepting new tool calls.
2. Drain the `create_draft` mutex — wait for any in-flight worktree creation to
   finish or roll it back (if the commit didn't complete, `git worktree remove`
   the half-built worktree and delete its branch).
3. Exit cleanly.
This prevents orphaned worktrees + branches when the agent crashes or restarts
mid-`create_draft`.

**Read tools (all return JSON; not-found returns `{found: false}` or an empty
array, never a thrown error, so agents can branch on absence):**
- `list_docs({ type?, status?, tag? })` → `[{slug, type, title, status}]`,
  glob + frontmatter filter.
- `search_docs({ query, type? })` → ranked `[{slug, title, snippet, score}]`.
  A doc matches if every whitespace-split term appears (case-insensitive) as a
  substring in `title || tags || body`. Scoring:
  `score = 3*(any tag match) + 2*(title substring match) + 1*(body match)`;
  sort descending, cap results at 20. Zero matches → empty array.
- `get_doc({ slug })` → `{found, slug, frontmatter, body}`. Non-unique slug is
  a `check` error, not a runtime ambiguity.
- `get_glossary_term({ name })` → `{found, slug, definition, related: []}`,
  or on a miss `{found: false, suggestions: [top-3 candidate slugs]}`. Matching
  order: exact slug → case-insensitive substring on slug/title → falls back
  to the `search_docs` scoring formula (tag/title/body weights) over `name`
  against all `glossary-term` docs, returning the top hit + its `related`.
  **(F9):** a miss must NOT return a confident wrong "closest" definition —
  that manufactures the exact hallucination RepoMind exists to reduce. A
  non-exact fallback returns `{found:false}` + the top-3 candidate slugs as
  suggestions, never a fabricated definition.
- `explore_graph({ slug, depth? })` →
  `{nodes: [{slug, type, title}], edges: [{from, to}], maxDepthReached: n,
  truncated: bool, broken_links: [slug]}`. BFS over `related:` edges, default
  depth 3, visited-set cuts cycles. `maxDepthReached` = the deepest depth at
  which a node was actually discovered; `truncated` = true if any frontier was
  cut at the requested depth. **(F15)** MCP tools don't run `check`, so a
  dangling `related:` entry (target doc missing) is handled at runtime by
  skipping the missing neighbor and recording its slug in `broken_links` —
  never an error, never a silent empty.

**Write tool (ships after P2 kill-switch passes):**
- `create_draft({ type, title, body, related? })` →
  `{branch, path, slug, cleanupCommand}` on success, or a **typed MCP error**
  (C2) on failure. **Safe git model — does NOT touch the user's working tree
  or HEAD:** uses `git worktree add` into an isolated path
  (`.project-knowledge/.worktrees/<slug>`), writes the templated doc at the
  type-mapped path (see Type → directory mapping) on the branch, commits,
  returns the branch name + path + a `cleanupCommand`
  (`git worktree remove .project-knowledge/.worktrees/<slug>`) the user runs
  after merging the PR. The MCP server's cwd is never switched. `create_draft`
  calls are serialized behind an in-process mutex and retry once on a git
  worktree lock error, so concurrent calls never collide at the git level. For
  cross-process safety (two Claude Code sessions → two MCP servers, each with
  its own in-process mutex — F13), also take a file lock
  (`.project-knowledge/.worktrees/.lock` via `flock`) around the worktree-add
  critical section, so two server processes can't race the same git worktree
  lock.
  Branch name: `repo-mind/draft/<type>-<slug>-<short-uuid>`. **Path-collision
  rule (F6 — preserves the slug→path invariant):** if the target path
  exists, append `-2`, `-3`, … to BOTH the filename AND the frontmatter
  `slug` field written into the doc — so a doc written as `auth-2.md` is
  queryable via `get_doc("auth-2")`. (Suffixing only the filename would orphan
  the doc: `auth-2.md` would be unreachable via `get_doc("auth")` and via
  `get_doc("auth-2")` since its frontmatter slug is still `auth`.) If all
  suffixes up to `-9` are taken, return `WRITE_CONFLICT` rather than keep
  suffixing.
- **Typed error codes (C2):** `ENOENT_REPO` (no `.git` — "initialize git
  first"; `init` also warns if `.git` is absent), `LOCK_RETRY_EXHAUSTED`
  (worktree lock held after one retry), `WRITE_CONFLICT` (target path exists
  and all `-2`/`-3`/… suffixes exhausted), `BAD_TYPE` (`type` not in the enum),
  `BAD_SLUG` (slug fails `^[a-z0-9][a-z0-9-]*$`). Each is returned as a
  structured MCP error with a code + message, never a bare free-text string,
  so the agent can branch programmatically.
- **Template render (C3) — no template engine:** create_draft writes a doc by
  plain string assembly (no Handlebars/EJS dependency). It emits YAML
  frontmatter derived from args + defaults — `type`, `slug` (kebab of title),
  `status: proposed`, `title`, `related` (if given), `updated: <today's ISO
  date>` — followed by a blank line and the caller-supplied `body` verbatim,
  based on a per-type template file that holds only the frontmatter scaffold
  with placeholder values. Boring by default.

**`check` v1** (kept in the CLI): validates frontmatter schema, slug
uniqueness, and that every `related:` slug resolves to an existing doc. Also
warns about orphaned draft worktrees — entries under
`.project-knowledge/.worktrees/` whose branch has no merge in progress and is
older than 7 days (A4). Output is human-readable: one line per violation with
the file path and the problem. Exit code is non-zero if any *violation* is
found (orphaned-worktree warnings do not fail CI, only schema/uniqueness/
broken-link violations do). Works in CI and pre-commit hooks. No
staleness/owner/AI-review checks in v1 (those are `check` v2, deferred).

**`export` v1** — emits `agents.md` only, written to the repo root
(`./agents.md`). **(F14)** refuses to overwrite an existing `agents.md`
unless `--force` is passed (a hand-authored `agents.md` is never silently
clobbered). A single Markdown file with one
section per `type` (## ADRs, ## Feature Specs, ## Glossary, ## Open
Questions, ## Agent Instructions), each listing that type's docs by title +
slug + status, followed by the full body. Frontmatter is rendered as a fenced
YAML block at the top of each doc. This is the fallback path for agents that
read files directly; the MCP is the preferred path. (`llms.txt` and
`docs-index.json` deferred — see Open Questions.)

**In-process cache.** `Map<path, {mtime, frontmatter, body}>`. On each tool
call, re-glob the `.project-knowledge/` folder; for any path whose mtime
changed or that appeared/disappeared since last parse, re-parse that one file
only. No cross-process cache, no persistent state.

**Shared `DocIndex` module (C1 — DRY gate).** A single `DocIndex` module owns
knowledge-root discovery (A3), the in-process cache above, slug validation
(A1), and frontmatter parsing. All five read tools, plus the `check` and
`export` CLI commands, consume this one module — no tool globs or parses
frontmatter independently. This is the DRY invariant: discovery + cache +
slug-safety + parse live in exactly one place; drift across tools is
structurally impossible. The MCP server holds one `DocIndex` instance for its
lifetime; each CLI command constructs its own (process-scoped).

**Recommended package layout (convention, not a contract):**
`src/cli.ts` (command dispatcher), `src/mcp/server.ts` (MCP stdio server +
lifecycle handlers), `src/index/doc-index.ts` (the shared module), `src/git/
worktree.ts` (create_draft's git ops + mutex), `src/tools/<tool>.ts` (one file
per tool, each a thin handler over DocIndex), `templates/<type>.md` (the five
per-type frontmatter scaffolds). `tests/` mirrors `src/tools/` + an `ab-demo/`
fixture.

**Performance trigger (P1) — when to upgrade to Approach B.** v1's in-memory
cache + O(n) grep search is fine at hundreds of docs (sub-ms search, a few MB).
Implement Approach B (SQLite read index, rebuildable from files) when ANY of:
**>1000 docs**, OR **p95 `search_docs` >200ms** on the largest repo tested, OR
**cache >50MB**. This turns the scale path from a vague "slow at thousands"
into a measured, actionable milestone.

## Test Plan

Framework: **vitest** (greenfield TS — added to `devDependencies` at scaffold).
Goal: 100% coverage of traced codepaths + user flows; too-many-tests > too-few
per the builder's stated preference. The full per-branch list lives in the
standalone test plan artifact
`york-repomind-eng-review-test-plan-20260621.md` (consumed by `/qa`); this
section commits to the coverage shape.

**Unit tests (one file per tool + DocIndex), `tests/`:**
- `doc-index.test.ts` — discoverRoot walk-up (found / not-found), cache mtime
  (unchanged/changed/new/deleted), slug validation (valid + the traversal
  suite: `../x`, `a/../../b`, `..%2f..`, absolute, empty, `/`-containing),
  prefix-check (under-root vs escapes-root). This is the **security gate A1**
  and must be a dedicated adversarial suite asserting zero filesystem access
  outside `.project-knowledge/` for every bad slug.
- `list_docs.test.ts` — no-filter / type / status / tag / AND-combine / empty→[].
- `search_docs.test.ts` — empty/whitespace q→[], single term, multi-term AND,
  no-match→[], >20 capped + sorted.
- `get_doc.test.ts` — found, not-found→{found:false}, bad-slug, non-unique.
- `get_glossary_term.test.ts` — exact/substring/fallback/not-found,
  definition=first paragraph, related returned.
- `explore_graph.test.ts` — depth-3 default, depth≤0→clamp to 1 (root only),
  cyclic→visited-set, truncated flag, isolated node.
- `create_draft.test.ts` — typed errors (BAD_TYPE/BAD_SLUG/ENOENT_REPO/
  LOCK_RETRY_EXHAUSTED/WRITE_CONFLICT), path-exists -2/-3, success path
  (worktree+branch+commit, cleanupCommand returned), mutex serializes
  concurrent calls.
- `check.test.ts` — schema/dup/broken-related violations → non-zero exit;
  orphan-worktree warning → exit 0 (warning doesn't fail CI).
- `export.test.ts` — empty→minimal agents.md, overwrite, per-type sections,
  fenced frontmatter.
- `init.test.ts` — scaffold, no-.git warn, idempotent (re-run doesn't clobber).
- `server-lifecycle.test.ts` — SIGTERM/SIGINT/stdin-close mid-create_draft →
  drain mutex, rollback half-built worktree, exit. No orphaned worktree.

**E2E tests (`tests/e2e/`):**
- `install-to-whoa.e2e.ts` — npx init → MCP config → agent question → grounded
  answer, <5 min on the demo repo.
- `create-draft-lifecycle.e2e.ts` — create → human PR review → merge →
  `cleanupCommand` → worktree removed; assert the user's working tree/HEAD
  were never touched. **Regression-critical**: a future refactor must not
  regress to `git checkout -b`.
- `concurrent-create-draft.e2e.ts` — two parallel calls → mutex serializes →
  two distinct branches, no git lock error.
- `run-from-subdir.e2e.ts` — MCP launched from a subpackage → walk-up finds
  root → tools return correct results (not silent empty).

**EVAL harness (`ab-demo/`, the P2 kill-switch — NOT a unit test):**
- `run-ab.ts` — scripted, reproducible. Seeds ~15 docs, runs 5–10 diverse
  questions twice (plain markdown+CLAUDE.md vs RepoMind MCP), captures median
  tokens-in-context (from agent usage metadata) + total hallucination points
  (human, 3-point rubric, blind to run). Output: a JSON results file + the
  README hero stat. **If RepoMind shows no visible gap, STOP (per P2).**

**Coverage target:** every branch in the coverage diagram traced in the
eng-review test plan artifact has a test; E2E covers the 3 cross-component
flows; EVAL covers the kill-switch. No untested branch ships.

## Open Questions

1. **Does retrieval actually beat plain markdown?** The A/B demo answers this.
   If the token/hallucination gap is not visible, stop. Unresolved until the
   demo runs — this is the kill-switch, intentionally not pre-answered.
2. **Is grep-ranking good enough?** The scoring formula above is the v1 plan.
   The demo surfaces whether agents prefer `search_docs` over reading files;
   if ranking is too rough, embeddings/SQLite FTS (Approach B) is the upgrade.
3. **`export` breadth.** v1 ships `agents.md` (the most agent-ecosystem-
   standard format) as the single export. `llms.txt` and `docs-index.json`
   are added only when a user asks — one format proves the safety-net thesis,
   three is YAGNI until demand appears.

## Success Criteria

- **P2 kill-switch passes:** In the A/B demo on a seeded ~15-doc repo, the
  RepoMind-MCP run beats the baselines on a **falsifiable, pre-registered
  experiment** (per outside-voice rigor fixes F1–F5), across **15, 50, and 200
  docs** (not a single scale — the thesis is most likely false at small
  corpus because the 6 MCP tool-definition schemas sit permanently in the
  agent's context, so a 15-doc CLAUDE.md can be smaller than the tool defs
  themselves). The experiment MUST be falsifiable:
  - **Pre-registration:** the 5–10 diverse project questions are written down
    and committed BEFORE any run; a second person (not the builder) writes
    half the questions.
  - **Independent baseline:** the plain-markdown + CLAUDE.md baseline is
    authored by someone other than the builder (the builder has incentive for
    RepoMind to win; a self-authored weak baseline is theater).
  - **Three arms:** (a) plain markdown + CLAUDE.md, (b) RepoMind MCP, **(c)
    the builder's actual gstack setup** — P3 names gstack as the real
    competitor, so the demo must test against it, not a strawman. If RepoMind
    loses to gstack, kill or pivot.
  - **Right metric:** total tokens consumed per answered question (input +
    output across the full turn, INCLUDING all MCP round-trips
    `search_docs`→`get_doc`→`explore_graph`), not just tokens-in-context
    (which rewards RepoMind by construction — smaller payloads is the design).
    Report median total tokens across the question set.
  - **Numeric pass bar (pre-decided):** ≥30% lower median total tokens AND a
    hallucination-point gap of ≥2 vs EACH baseline arm. "If the gap isn't
    visible, stop" is now judged against this bar, not the builder's feeling.
  - **Break-even corpus size:** report the corpus size at which RepoMind
    first beats each arm — if it's above the `init` scaffold size, the <5-min
    whoa (which runs on ~5 init docs) is unsupportable and the thesis is
    false at demo scale.
  - **Agent-preference steering (F5):** specify how agents are nudged to
    choose `search_docs` over `cat README.md` — `init` injects a CLAUDE.md
    snippet ("prefer repo-mind search_docs over reading docs/ directly"),
    and tool descriptions are written to steer. This steering is part of the
    experiment, not assumed.
  - **Ranker isolation (F8):** the v1 grep ranker (tag +3 / title +2 / body
    +1) confounds the kill-switch — a known-rough ranker means a failed demo
    can't distinguish "thesis false" from "ranker bad." Run the demo with an
    oracle ranker (manually-curated correct result per question) as a control
    arm, so a RepoMind-with-oracle failure isolates the thesis from the
    ranker. If RepoMind-with-oracle still loses, the thesis is false.
  Hallucinations are scored by a human on a 3-point rubric per answer
  (0 = none, 1 = minor, 2 = major), blind to which arm produced it; report
  total hallucination points per arm. The (median total tokens, hallucination
  points per arm, break-even corpus size) tuple is the README hero stat.
- **<5-minute install-to-whoa:** From `npx repo-mind init` to an agent
  successfully answering a project question via the MCP, under 5 minutes, on
  the demo repo. Verified by running the acceptance test before shipping.
- **Five read tools work end-to-end** over frontmatter: list, search, get,
  glossary, explore (`create_draft` ships after the kill-switch passes, per
  P1).
- **`init` scaffolds the convention** with five generic templates (ADR,
  Feature Spec, Glossary Term, Open Question, Agent Instruction) and the
  type-mapped subdirectory layout.
- **`export` emits `agents.md`** — the fallback path for agents that read
  files directly, proving the MCP is the preferred path and the export is the
  safety net. (`llms.txt` and `docs-index.json` deferred — see Open Questions.)

## Why not gstack? (build vs contribute — F20)

The outside voice's hardest strategic question: the builder already runs gstack,
which has design docs (`~/.gstack/projects/`), learnings, decisions, and project
structure that RepoMind reimplements in weaker form. Why build a parallel tool
competing for the same niche instead of contributing an MCP layer over gstack's
artifacts?

The boundary that answers it: **gstack is developer-centric and cross-repo;
RepoMind is repo-centric and lives inside the repository.** gstack's knowledge
lives in `~/.gstack/` (the developer's machine, per-developer, not versioned
with the code) and is oriented around the *developer's workflow* (sessions,
skills, design docs). RepoMind's knowledge lives in `.project-knowledge/`
*inside the repo* — versioned with the code, shared by every agent and human
that clones it, scoped to the project, not the developer. A team member or a
fresh CI agent cloning the repo gets the knowledge for free; gstack's
`~/.gstack/` never travels with the clone. That's the load-bearing difference:
**knowledge that survives the clone is the product.** gstack can't be that
without moving its artifacts into the repo, which would change what gstack is.

So: RepoMind is not "gstack but weaker" — it's the *repo-resident* knowledge
layer that gstack structurally isn't. The right long-term move may be to let
gstack *read* a repo's `.project-knowledge/` (a gstack MCP that queries
RepoMind's convention), unifying the developer-centric and repo-centric layers
without either reimplementing the other. v1 ships RepoMind standalone; the
gstack-bridge is a roadmap item, not a reason not to build.

## Reviewer Concerns

Minor edge-case gaps surfaced by the adversarial spec review (3 iterations,
final quality score 8/10). None are structural; each is a one-line
implementation decision to nail down during the build:

- **`list_docs` filter combination:** `type?`, `status?`, `tag?` filters
  AND-combine; an omitted filter is a wildcard.
- **`search_docs` empty query:** empty or whitespace-only `query` → empty
  array (no error).
- **`init` template bodies:** the frontmatter schema is specced; the template
  *body* content is implementer-discretionary per the schema (each example
  doc should be realistic enough to feed the A/B demo).
- **`explore_graph` depth bounds:** depth is clamped to ≥ 1; `depth: 0` (or
  negative) is treated as 1 (root node only, no edges).
- **mtime cache precision:** the in-process mtime cache can theoretically go
  stale on sub-second same-mtime edits — rare and negligible, but the
  implementer should not assume mtime is a perfect invalidation signal; a
  `repo-mind check --rebuild` or process restart clears it.
