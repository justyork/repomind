## Problem Statement

RepoMind v1 ships MCP read tools and CLI over `.project-knowledge/`, but humans still edit markdown in an editor and visualize `related:` links only in their head. The approved v1 design explicitly deferred Web UI and SQLite. The product vision in `idea.md` positions **local Web UI** as a first-class surface: browse, search, graph, edit drafts, publish to git.

The user wants UI as the **primary human-facing differentiator** — not "another MCP server," but a **Notion-like workspace that lives in the repo** and stays in sync with agents via MCP.

**Status quo today:** Developers with AI agents use `docs/` folders, Notion, or Obsidian. Agents query via RepoMind MCP or ingest whole files. No single surface shows graph + draft/publish + agent-readable canonical markdown.

## What Makes This Cool

**The "whoa" moment:** Open `repo-mind ui`, see your project's knowledge as a **living graph** (ADRs, glossary, specs linked by `related:`), click a node, edit a draft in the sidebar, hit Publish — and the agent in Cursor instantly sees the updated doc via MCP on the next query. Human and agent share one knowledge layer; git is the merge gate.

Secondary delight: **split view** showing "what the human sees" (rendered markdown + graph) vs "what the agent gets" (MCP `get_doc` JSON) for the same slug.

## Constraints

- **Local-only** — UI binds `127.0.0.1`, no auth in v1 (per `idea.md` §19).
- **Git remains source of truth for published knowledge** — SQLite holds drafts only until Publish.
- **Do not break MCP v1 contracts** — published files under `.project-knowledge/` remain plain markdown + YAML frontmatter; DocIndex is shared.
- **P2 A/B kill-switch runs in parallel** — UI does not block A/B harness work (premise P5).
- **TypeScript + Node** — same package as CLI; minimize new native deps (SQLite: `better-sqlite3` acceptable for UI milestone; document in README).
- **Russian + English content** — UI chrome in English; docs can be any language (skyforge dogfood).

## Premises (agreed in session)

1. **P1:** Hero screen = interactive **knowledge graph** from `related:` frontmatter (`explore_graph` semantics).
2. **P2:** **SQLite draft layer** — UI writes drafts; **Publish** converts to `.project-knowledge/*.md` in git.
3. **P3:** Position vs **Notion** — win on in-repo, git-native, MCP-native, no cloud lock-in.
4. **P4:** Primary user = **developer with AI agent** (Cursor / Claude Code).
5. **P5:** UI milestone **parallel** to P2 A/B demo, not gated on it.

## Landscape (Phase 2.75)

| Layer | Conventional wisdom |
|-------|---------------------|
| L1 | Notion = team wiki; Obsidian = personal graph + markdown; `docs/` in git = engineer default |
| L2 | 2026 entrants: **Medulla** (CRDT + MCP + snapshots), **gitmark** (markdown + FTS + HTML graph), **Lumen/Cognexia** (SQLite + MCP + graph) |
| L3 | **EUREKA:** Market splits "markdown in git" OR "SQLite brain." RepoMind's wedge is **triple intersection**: Notion-feel UI + git publish + MCP query — few tools own all three |

**Why not Obsidian?** No in-repo convention, no MCP-first agent contract, no team publish flow to git.

**Why not Notion?** Knowledge outside repo, no agent MCP, no PR-based truth.

## Approaches Considered

### Approach A: Graph Weekend (read-only)

- `repo-mind ui` — static server + force-directed graph + search sidebar
- Reuses `DocIndex`, `explore_graph`, `search_docs`
- **Effort:** S (human: 1–2d / CC: ~45m) | **Risk:** Low
- **Pros:** Fast demo; proves graph UX
- **Cons:** No edit; weak Notion positioning

### Approach B: SQLite Draft + Publish (chosen)

- UI → SQLite drafts → Publish → markdown files
- Graph hero + editor + status dashboard + `check` integration
- **Effort:** XL phased (human: 2–3w / CC: ~3–4h) | **Risk:** Med–High
- **Pros:** Matches `idea.md`; clear differentiation
- **Cons:** Scope; draft/publish state machine; SQLite dependency

### Approach C: Graph-first Direct Edit

- Edit markdown files directly; optional SQLite as read index only
- **Effort:** M | **Risk:** Med
- **Pros:** Simpler than B; git-native
- **Cons:** No draft vs published UX like Notion

## Recommended Approach

**Approach B in three phases** — ship graph wow early, add SQLite when phase 2 starts.

### Phase UI-1: Graph Workspace (read-only) — ~weekend

**Command:** `repo-mind ui [--port 3847] [--cwd]`

**Stack:**
- `src/ui/server.ts` — Node `http`, REST API
- `ui/` — Vite + vanilla TS or lightweight Preact (no heavy framework)
- Graph: **d3-force** or **vis-network** (pin version)

**Screens:**
1. **Graph view (default)** — nodes = docs; edges = `related:`; color by `type`; size by link degree; click → detail panel
2. **Sidebar** — filter by type/status/tag; search with `search_docs` ranking
3. **Doc panel** — rendered markdown preview + frontmatter YAML block + "Open in editor" (external) stub
4. **Agent preview tab** — pretty-printed JSON from `get_doc` / `get_glossary_term` for same slug

**API (read-only):**
```
GET  /api/health
GET  /api/docs?type&status&tag
GET  /api/docs/:slug
GET  /api/search?q=&type=
GET  /api/graph/:slug?depth=
GET  /api/stats        → counts by type/status, broken related count from check
```

**Acceptance:** Graph renders skyforge-caravan `.project-knowledge/` (10+ nodes); search finds `caravan`; localhost only.

### Phase UI-2: SQLite Draft Layer — ~1 week

**Storage:** `.project-knowledge/.repo-mind/drafts.db` (gitignored)

**Schema (minimal):**
```sql
CREATE TABLE drafts (
  id TEXT PRIMARY KEY,           -- uuid
  slug TEXT NOT NULL,
  type TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'draft',
  title TEXT,
  body TEXT NOT NULL,
  tags_json TEXT DEFAULT '[]',
  related_json TEXT DEFAULT '[]',
  published_path TEXT,           -- null until published
  created_at TEXT,
  updated_at TEXT,
  UNIQUE(slug) WHERE published_path IS NULL  -- one active draft per slug
);
CREATE TABLE publish_log (
  id TEXT PRIMARY KEY,
  draft_id TEXT,
  git_path TEXT,
  published_at TEXT
);
```

**Rules:**
- **New doc** — create draft from template (`templates/*.md`); slug validated A1
- **Edit published doc** — fork to draft (copy frontmatter + body); show badge "draft of `slug`"
- **Autosave** — debounced PUT to SQLite (not git)
- **Publish** — validate via `check` rules → write `TYPE_TO_DIR/<slug>.md` → invalidate DocIndex cache → optional `git status` hint in UI

**API (write):**
```
POST   /api/drafts              → create
PUT    /api/drafts/:id          → autosave
POST   /api/drafts/:id/publish  → write markdown + return path
DELETE /api/drafts/:id          → discard draft
GET    /api/drafts              → list unpublished
```

**UI additions:**
- Split editor: frontmatter form fields + markdown body textarea + live preview
- Status chips: draft | proposed | accepted | superseded
- Publish button → confirmation modal showing target path

### Phase UI-3: Publish Flow + Dashboard — ~1 week

- **Dashboard** — outcome of `repo-mind check`: schema errors, duplicate slugs, broken `related:`, orphan worktrees (warnings)
- **Publish queue** — list drafts ready to publish; batch publish (optional)
- **Diff preview** — before publish, show unified diff vs current file on disk
- **Export shortcut** — trigger `agents.md` export from UI
- **Future hook:** `publish --pr` (branch + commit message template) — design only, implement v1.1

### MCP integration (unchanged contract)

- MCP reads **published** markdown only (DocIndex refresh on mtime)
- Drafts invisible to MCP until Publish — **intentional** (agents must not see unreviewed knowledge)
- Optional v1.1: MCP tool `list_drafts` read-only for agent awareness (out of scope UI-3)

## Open Questions

1. **SQLite vs libsql/wasm** — `better-sqlite3` is simplest for local CLI; revisit if Windows ARM blocks adopters.
2. **Monorepo UI package** — single package vs `packages/ui` — stay single package until UI exceeds ~3k LOC.
3. **Real-time collab** — explicitly out (Notion parity not required v1).
4. **Import from `docs/`** — one-shot migration wizard? Defer unless dogfood demands it.

- Shipped in **same npm package** `repo-mind` — `npx repo-mind ui`
- UI assets bundled in `dist/ui/` (no CDN)
- CI: add `npm run build:ui` + smoke test `curl /api/health` after `ui` start
- GitHub Releases unchanged (no separate binary)

## What I noticed about how you think

- You chose **knowledge graph** as the hero screen even though MCP already has `explore_graph` — you care about **human spatial understanding**, not just agent APIs.
- You picked **Notion** as the competitor, then **SQLite + publish** over the weekend shortcut — you're willing to pay complexity for the **draft/review** story, not just a pretty viewer.
- You kept **P2 A/B parallel** — you haven't abandoned the kill-switch thesis while expanding scope; that's discipline.
- Hackathon framing with **2+ week scope** — you want a demo that can become a product, not a throwaway prototype.
